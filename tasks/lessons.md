## architecture — 設計・責務・config

### [知識拡張と構造改善の分離]
- **症状**: 検索構造の改善に変化がないと、Wiki全体の成長も停止していた。
- **原因**: 既存ページ改善を優先し、新しい関連知識を追加するタスクが不足していた。
- **対策**: `expand_knowledge`を独立した主要タスクとして扱い、構造改善の結果に関係なく、未調査の関連情報を新規ページとして蓄積する。

### [`reflections`テーブルが存在するのに一度も書き込まれていなかった]
- **症状**: SQLiteスキーマに`reflections`テーブル（`run_id, problem, lesson, proposed_rule`）が定義されていたが、`run_once()`内のどの失敗パスからも`INSERT`されておらず、実行するほど失敗の経験が蓄積されるはずの仕組みが機能していなかった。加えて調査の過程で、`create_structure`/`expand_knowledge`のReviewer拒否パスは`runs`テーブルへの記録自体も欠落していたことが判明した（`create_page`/`improve_page`側には存在した）。
- **原因**: 機能追加のたびに個別の失敗ハンドリングを書き足しており、失敗イベントを横断的に記録する仕組みが整備されないまま進んでいた。
- **対策**: `StateDB.record_reflection()`を追加し、`plan_rejected`・両経路のReviewer拒否・`push_failed`の各失敗イベントで呼び出すようにした。`proposed_rule`はLLMによる追加合成を伴うため今回は`None`のまま保存し、範囲を「既存の失敗情報を記録する」ことに限定した。合わせて`create_structure`/`expand_knowledge`のReviewer拒否パスに欠けていた`runs`INSERTも追加した。

### [`pages.updated_at`が実行のたびに上書きされ鮮度判定が機能していなかった]
- **症状**: 鮮度管理機能（#7）を実装しようとしたところ、`StateDB.sync_pages()`が`updated_at`を常に`now()`で書き込んでおり、`run_once()`が呼ばれるたび（＝ページ内容が変わっていなくても）に「今更新された」扱いになっていた。このままでは「古いページ」を検出する仕組みが原理的に成立しない。
- **原因**: `sync_pages()`はVault上のファイルをDBに反映する処理として実装されただけで、「実際にいつ変更されたか」という観点が考慮されていなかった。
- **対策**: `updated_at`をファイルの実際のmtime（`path.stat().st_mtime`）に基づく値へ変更した。これによりファイルが実際に書き換えられた時だけ`updated_at`が進むようになり、`StateDB.stale_pages(days)`で閾値より古いページを正しく検出できるようになった。

## quality — テスト・CI/CD・品質保証
### [サブカテゴリ: タイトル]

### [uvキャッシュ権限]
- **症状**: ユーザー領域のuvキャッシュへ書き込めず検証が開始できなかった。
- **原因**: 実行環境のキャッシュディレクトリ権限が制限されていた。
- **対策**: プロジェクト内の`.uv-cache`を`UV_CACHE_DIR`に指定して実行する。

### [実Reviewerによる安全停止]
- **症状**: Writerが生成したページを実Reviewerが不合格と判定した。
- **原因**: プレースホルダー、出典不足、未解決点の残存があった。
- **対策**: Reviewer合格前はVaultへ書き込まず、`review_rejected`と理由をSQLiteへ保存する。

### [Writer出力の二重エスケープでフロントマターが壊れる]
- **症状**: 実Ollama実行でVaultに書き込まれたページのフロントマターが `---\ntype: knowledge\n...` のようにリテラルな`\n`文字列のまま保存され、Markdownとして機能しなかった。
- **原因**: モデルがネストしたJSON文字列を二重エスケープして返すことがあり（`content`フィールド内に実改行ではなくリテラル`\n`が入る）、`Ollama.write()`がそれをそのままVaultへ書き込んでいた。
- **対策**: `unescape_literal_newlines()`を追加し、実改行が皆無で`\n`リテラルのみ含む場合だけアンエスケープする（日本語本文を壊す`unicode_escape`は使わない）。回帰テストで固定。

### [Reviewerのtype:blockingを無視してVault書き込みが素通りしていた]
- **症状**: 実Reviewerが`issues`に`"type": "blocking"`と明示していても、内容の薄いページがそのままVaultに書き込まれた（`review_warnings`は返るが`result: success`扱い）。
- **原因**: `review_is_blocking()`が固定キーワードの文字列一致だけで判定しており、モデル自身が返す構造化された`type`フィールドを見ていなかった。
- **対策**: `issues`内の各要素に`type == "blocking"`があれば即ブロックするよう修正し、Reviewerへのシステムプロンプトも`{"type": "blocking"|"warning", "description": ...}`の形式を明示するよう変更した。

### [Writerがコードフェンスでページ全体を包むとフロントマターが二重になる]
- **症状**: モデルが応答全体を \`\`\`markdown ... \`\`\` で包んで返すことがあり、`normalize_page()`が先頭を`---`と誤認識できず、独自のフロントマターを追加でページ先頭に足してしまい、Vault内のページが二重フロントマター＋ぶら下がったフェンス残骸になった。
- **原因**: Writer出力に対してコードフェンスの除去処理がなかった。
- **対策**: `strip_markdown_fence()`を追加し、応答全体が単一の\`\`\`フェンスで囲まれている場合のみ中身を取り出してから`normalize_page()`に渡す。

### [`Git`がVaultではなくソースリポジトリをコミット対象にしていた]
- **症状**: `Git(Path.cwd())`と実装されており、`auto_commit`有効時にWiki本体（`config.vault_path`）ではなく、エージェントのソースコード自体が置かれているリポジトリをコミット対象にしてしまう設計になっていた。
- **原因**: `run_once`内で`Git`クラスへ`Path.cwd()`を渡していたため。Vaultがソースリポジトリの外（別のObsidian Vaultフォルダ）にある運用では致命的な誤動作になる。
- **対策**: `Git`の初期化を`vault.root`に変更し、`is_repo()`を追加してVaultがgit管理下にない場合は安全に何もしないようにした。

### [新規ページ作成前の重複チェックが存在しなかった]
- **症状**: 実Ollama検証で、同じ「ホームページ作成」という作業を複数回実行した際、モデルが毎回微妙に異なるタイトル（Home / ホームページ 等）を提案する可能性があり、対策がないと類似ページが量産される。仕様書§6.1が要求する「類似タイトル・同義語の確認」が未実装だった。
- **原因**: `choose_candidate`/`Ollama.plan`が返した`target`をそのまま`create_page`していた。
- **対策**: `find_similar_page()`（正規化した完全一致・部分文字列一致・文字bigram類似度）を追加し、類似ページが見つかった場合は新規作成せず既存ページへの`improve_page`にリダイレクトするようにした。

### [ロールバック機能は見送り]
- **症状**: ユーザーからの機能要望リストに「変更前後の比較とロールバック」が含まれていたが、そのまま実装しなかった。
- **原因**: 要求の本質は検索品質の前後比較に基づくロールバックであり、単純な「例外時にファイルを元に戻す」実装ではこの要求を満たさない。また、別プロセスが`experiments/retrieval_experiment.py`等で検索品質のベンチマークに取り組んでいる最中であり、重複実装・後の衝突のリスクがあった。
- **対策**: 実装を見送り、他プロセスの検索実験が形になった段階で改めて設計を検討することにした（[[自律Wiki構築AI]]参照）。

### [コミット条件が反転していてauto_commitが実質発火しなかった]
- **症状**: `if vault_git.is_repo() and not vault_git.status(): commit()` となっており、「状態がクリーン（＝差分なし）」の時だけコミットする条件になっていた。直前に`vault.write()`で新しい内容を書き込んだ直後は必ず差分が生じるため、この条件はほぼ常にFalseになり、`auto_commit`を有効にしても実際にはコミットが発火しない状態だった。
- **原因**: 「差分があるときにコミットする」の条件を`not status()`と書き間違えていた（元の実装から存在していた）。また`create_structure`/`expand_knowledge`の成功パスにはコミット処理自体が存在しなかった。
- **対策**: `commit_and_push()`として条件を`vault_git.status()`（差分がある場合）に修正し、`Git.status()`もVaultディレクトリへのpathspec（`--  <root>`）でスコープすることで、リポジトリ内の無関係な変更（同じリポジトリで並行作業している別プロセスの未コミット差分等）に反応しないようにした。全ての書き込みパスから`commit_and_push()`を呼ぶよう統一した。

### [qwen3のthinkingによる実行時間超過]
- **症状**: qwen3:8bが単純なJSON応答でも長いthinkingを生成し、Planner単体がタイムアウトした。
- **原因**: 構造化された短い判断タスクにも思考出力が有効化され、`keep_alive: 0`によるモデルロード時間も各呼び出しに加算された。
- **対策**: Ollama Chat APIへ`think: false`を指定し、thinkingを使わない構造化タスクとして実行する。改善しない場合は軽量モデルとの比較へ進める。

### [auto_push有効化直後、pushの衝突でrun全体がクラッシュする設計だった]
- **症状**: `auto_push`を有効化した直後に想定される事態として、この同じリポジトリへ並行して書き込んでいる別プロセスが先にpushしていた場合、`git push`がnon-fast-forwardで拒否され、`Git.push()`の`check=True`がそのまま例外を投げてrun全体を失敗させてしまう設計だった。ローカルの知識自体は正しく保存されているのに、push層の問題だけでrunが失敗扱いになるのは過剰な失敗判定である。
- **原因**: `subprocess.run(..., check=True)`で単純にpushしているだけで、リジェクト時のリトライ・非致命化処理がなかった。
- **対策**: `Git.push()`を「1回失敗したら`fetch`＋`rebase`を試し、それでも失敗したら例外を投げず`False`を返す」形に変更。`commit_and_push()`の戻り値を`"skipped"/"committed"/"pushed"/"push_failed"`の文字列にし、`run_once()`の結果へ`git_status`として記録するようにした。push失敗はrun失敗として扱わない。

### [strip_markdown_fenceが「閉じフェンス無し」「閉じフェンス後の余剰テキスト」に対応できていなかった]
- **症状**: `auto_commit`有効化後の実運用（`538523e`）で、`normalize_page()`がWriter出力を`---`始まりと認識できず、二重フロントマターになる不具合が再発した。原因を辿ると、Writerの応答が \`\`\`markdown で始まったまま**一度も閉じフェンスを出力していなかった**（以前の事例は閉じフェンスの後に余分なテキストが続くパターンだったが、今回は閉じフェンス自体が存在しなかった）。
- **原因**: `strip_markdown_fence()`が「開きフェンスと閉じフェンスが両方揃って文字列全体を完全に囲んでいる」ケースしか検出できず、`re.match(...\`\`\`\s*$)`が閉じフェンスの欠落時にマッチせず、無変換のまま返していた。
- **対策**: 開きフェンスと閉じフェンスを独立に検出するよう書き直した。開きフェンス行があれば必ず除去し、閉じフェンス単独行（本文中のどこにあっても最後に現れたもの）が見つかればそこで本文を打ち切り、見つからなければ開き以降を全て本文として扱う。これにより「閉じフェンス無し」「閉じフェンス後の余剰テキスト」の両方に対応した。
